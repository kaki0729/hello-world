ソフトウェア開発の難しいところとしてソフトウェア開発に携わった経験がないと、イメージしにくいことや、知識を持っていても実践することが難しいこと、個々人のソフトウェア工学の知識だけでは不十分なためチーム全体で共有する必要があることなどが挙げられる。  
ソフトウェアは色々な機能単位に取り換えたり、追加することができる。そのため、ソフトウェアを取り替えることによって、ハードウェアを改変することなく、いろいろな機能を実現することができる。  
ソフトウェア工学の目標としては、ソフトウェアの難しさを払拭して生産性と品質を制御することである。また、ソフトウェア工学における最終的な目標として、品質、コスト、納期の三つの最適なバランスを実現するための手法や方法論を学ぶことである。  
ソフトウェア工学の必要性としては、大規模化と複雑さがどんどん増していくことや、求められるIT人材が増え続けること、社会性と求められるミッション、不確実性の拡大といった点が挙げられる。  
ソフトウェアライフサイクルはソフトウェアが誕生してから、稼働し、そしてその役割を終えて廃棄されるまでの一連の流れのことを指し、このときに大事になってくるのが要件定義である。  
要件定義とはシステム開発において、実装すべき機能や満たすべき性能などを明確にする作業のことで、プロジェクト関係者のバイブルや社内外への宣言と協力依頼などの意義がある。  
設計では要件定義書から設計書にし、コーディングができるレベルの仕様書に落とし込むことが大事である。また、プロジェクト目標を達成し、必要な要素成果物を生成するために、プロジェクトチームが実行する作業を、要素成果物を主体に階層的に要素分解したものであるWBS（Work Breakdown Structure）も設計によく使われている。  
WBSはスコープを明確にし、大きな作業をグルーピングした後、グループングした作業の相互関連を考え、各グループの作業を洗い出すことで作成される。  
製作するにあたっては、賃金の安い海外企業に外部委託するオフショア開発といったものがあり、日本のIT企業はオフショア開発される側になるのではないかといった懸念点がある。  
テストとデバッグにおいてはあらかじめテストしバグを早期に発見・対処することが理想であるが、全てをテストすることは不可能であるため、ソフトウェアのどの箇所をテストするか、どの程度テストをするかを決めるのが重要である。  
運用・保守については稼働後のデータをプロアクティブに活用するべき、ソフトウェア以外の要因も考慮する必要がある、開発者はプロジェクトから離れている可能性があるなどの懸念点がある。  
プロジェクトとは、明確な目標と期限が定められた、一時的な活動であり、有期性や独自性などの特徴がある。  
プロジェクトにおいて大切なこととして、フォアキャスティング・バックキャスティングといったものがある。フォアキャスティングは、今の立ち位置から順に目標に進んでいくものであるが、目標に到達できない可能性があることや目標と遠いところに到達する可能性があるなどの懸念点が挙げられる。バックキャスティングとは、定めた目標に向けて何が必要かを計画し達成に向かうものであるが、目標を明確に設定する必要があることや目標に向けての筋道を検討する必要があるなどの懸念点が挙げられる。  
ソフトウェアの評価においては、コードの物量が大事であり、組み込み型ソフトウェアはサイズが小さいほどよい。また、ファンクションポイント法といったものも存在し、FP値＝基準値×（0.65+調整値/100）、原価＝FP値×作業単価×換算値として、システム規模を概算している。  
開発プロセスでは、ウォーターフォール型開発プロセス、反復型開発プロセス、アジャイルプロセスなどがある。ウォーターフォール型開発プロセスでは進捗管理が容易、成果物が明確である一方後工程にしわ寄せが集中というリスクが存在している。反復型開発システムでは、部分的に完成させていくので、顧客の要求を取り入れやすい一方で、分割のための作業や管理業務が増えたり、全体像が見えづらいなどのデメリットが存在する。アジャイルプロセスではウォーターフォール型開発プロセスで想定されていなかった、できるだけ決定を遅らせることやできるだけ早く提供することを実現しているが、人的リスクなどの面もある。
gitのコマンドには様々なものがある。  設定や確認をするためのgitコマンドは以下のようになる。  
git init : gitの初期化や設定をする  
git status : ワークツリーのステータスを表示する  
git config : 設定周りの確認や変更をする  
git log : ログを表示  
git diff : ファイルの差分を表示する  
  コミットなどをするgitコマンドは以下のようなものが挙げられる。  
git add : ステージングエリアに追加する  
git commit : コミットの実行をする  
  修正をするためのgitコマンドは以下のようになる  
git commit --amend : コミットを修正する  
git checkout : 削除されたファイルを復旧したり、過去コミットを復元したりする  
git revert : コミットの変更を打ち消すコミットをする  
git reset : コミットをリセットする  
git rm : ファイルとindex情報を削除する  
  リモート系のgitコマンドは以下のようになる。  
git clone : レポジトリをコピーする  
git pull : リモートレポジトリを同期する  
git push : 変更をアップロードする  
git request-pull : プルリクエストをする  
git remote : リモートレポジトリの設定をする  
  ブランチ系のgitコマンドは以下のようになる。  
git branch : ブランチを作成する  
git checkout : ブランチを切り替える  
git merge : ブランチを統合する  
リモートレポジトリのホスティングサービスの一つとしてgithubがあり、オープンソースソフトウェアの主要なポータルサイトである。githubには、issuesといったレポジトリに関する問題や課題、機能追加などを挙げるための機能であるissueや、issueを管理するためのテーブルであるprojectなどがある。  
ソフトウェアの開発手法としてCI/CDといったものがある。  
CIはコード変更を共有リポジトリに頻繁に統合するプロセスであり、自動テストとビルドを定期的に実行するため、バグを早期発見して修正をすることができるようになる。CIが自動化することで開発においてスムーズに進行することができる。  
CDはコード変更をテスト環境や本番環境に自動的にデプロイするプロセスであり、自動デプロイメントを組み込み、手動操作によるデプロイの必要性を排除することができる。CDによってユーザーフィードバックを迅速に反映することが可能である。  
CI/CDパイプラインの基本的なステップは、ソース->ビルド->テスト->デプロイ->検証->モニタリングといった感じとなる。  
github Pagesはgithubにあるリモートレポジトリの内容をwebpageとして公開することができるシステムである。ホームページの公開はCI/CDにより自動化されており、Jekyllを使用することでmarkdownファイルを自動でhtml化することができる。そのため、開発者はmarkdownファイルを作成するだけで良いため、簡単にwebpageを作成できることができる。
